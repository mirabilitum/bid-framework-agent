# 分析任务

请从招标文件中提取以下信息：

## 1. 项目基本信息
- 项目名称（完整准确）
- 采购方式（公开招标/竞争性磋商/询比采购等）
- 预算金额（如有）

## 2. 包信息（如有多包）
- 识别文档中是否包含多个采购包
- 每个包的编号、名称、预算
- 包的关键字：包、标包、采购包、Package等
- **评分标准一致性判断**：如果多个包共用同一套评分标准（评分因素、分值、评分细则完全相同），在输出中标注 `"scoring_shared": true`；如果各包评分标准不同，标注 `"scoring_shared": false`

## 3. 评分标准（重点）
- 从评标办法、评审标准、评分因素等章节提取
- 每个评分因素包含：
  * 名称（如"培训服务方案"）
  * 分值（数字，如25.0）
  * 类别（technical技术/commercial商务/price价格）
  * 简要说明
  * 子项列表（如有，如["培训目标", "培训对象", "培训内容"]）
- 注意：价格分不需要展开子项
- 注意：识别父子评分结构（如"服务部分65分"包含多个子项）

## 4. 响应文件格式要求（核心：原封不动提取）

### 4.1 查找并原封不动提取"投标文件组成"

**核心原则**：找到招标文件中关于"投标文件组成"/"响应文件组成"的章节，**原封不动地提取其结构和文字**，不修改、不重组、不重新编号。

**查找位置**（按优先级）：
1. "投标文件组成"/"响应文件组成"/"投标文件应包括下述部分"章节
2. "投标文件编制要求"/"响应文件编制要求"章节
3. 如果以上都找不到，从"投标文件格式"章节中提取
4. 如果以上都找不到，检查"符合性审查"/"符合性检查/采购需求"/"比选需求"/"服务需求"等章节中查找是否有明确的多级标题结构（如"1、基础管理 → 1.1角色层级 → 1.2权限配置"），可作为skeleton来源
5. 如"符合性审查"/"符合性检查/采购需求"/"比选需求"/"服务需求"等章节中，有多个章节包含明确的多级结构，与评分表中的内容**综合比对**，取更完整、层级更清晰的版本作为skeleton

**提取要求**：
- **原封不动**：保留原文的编号格式（一、(1)、①、1.、格式一等）和标题文字
- **保留层级**：如果原文有嵌套结构，保留这个层级
- **不得重组**：不要按"资格/报价/技术"重新分组
- **不得遗漏**：每一项都必须提取
- **不得替换**：不要把概括性标题展开为具体子项列表

**常见结构类型**（供识别参考，不影响提取方式）：

**嵌套结构**（如福建格式）：
```
（1）资格及资信证明部分
  ①响应函
  ②供应商的资格及资信证明文件
（2）报价部分
  ①开标一览表
（3）技术商务部分
  ①标的说明一览表
  ②技术和服务要求响应表
```
→ 输出为嵌套结构：`[{"title": "（1）资格及资信证明部分", "items": ["①响应函", ...]}, ...]`

**扁平结构**（如江西格式）：
```
1.投标书
2.开标一览表
3.分项报价表
...
9.技术文件
10.其他资料
```
→ 输出为扁平列表：`["1.投标书", "2.开标一览表", ..., "10.其他资料"]`

**格式模板列表**（如韶关格式"格式一~格式N"）：
→ 如果文档中有独立的"投标文件组成"章节，以该章节为准
→ 如果没有独立的组成章节，格式列表本身就是骨架，原样输出

### 4.2 提取格式模板内容

如果招标文件有"投标文件格式"/"附件格式"等章节，提取每个模板的完整内容：

**必须提取完整正文，不是简要描述**。具体规则：

| 模板类型 | 提取内容 |
|---------|---------|
| 投标函/响应函 | **全文**（含所有承诺条款、签章处） |
| 承诺函/声明函 | **全文**（含所有承诺事项） |
| 授权书 | **全文**（含授权范围） |
| 中小企业声明函 | **全文**（含声明条款和表格） |
| 表格模板（开标一览表等） | **表头列名+列说明** |
| 封面格式 | **完整格式**（含所有字段） |
| "内容格式自拟"项 | 标注为自拟，不提取模板 |

**段落对齐标记**：提取full_text时，必须根据截图中的视觉排版保留原文的段落对齐方式，用行首标记标注：
- `[CENTER]文字` → 截图中居中的行（如函件标题"投标函"、"承诺函"、"授权委托书"等）
- `[RIGHT]文字` → 截图中右对齐的行（如签章区域"供应商：（盖章）"、"日期：年月日"、"法定代表人：（签字）"等落款行）
- 无标记 → 默认左对齐的行（正文段落）
- 缩进用空格表示：一级缩进2个空格，二级缩进4个空格，三级缩进6个空格（根据截图中的实际缩进程度判断级别）
- 表格部分用 `[TABLE_START]...[TABLE_END]` 包裹，列用 ` | ` 分隔（第一行为列头，后续行为数据）。根据截图判断哪些内容是表格形式

**视觉辅助判断**：截图是格式判断的主要依据。当文本内容无法确定对齐方式时，以截图中的视觉排版为准。

**提取格式**：
```json
[
  {
    "id": "格式编号（如有，如'格式一'、'格式7-1'、'(1)'等）",
    "title": "投标函",
    "content_type": "full_text / table_structure / cover_format / free_form",
    "full_text": "完整正文内容",
    "table_columns": "列名1 | 列名2 | 列名3（表格时使用）",
    "is_free_form": false,
    "notes": "附带说明"
  }
]
```

### 4.3 提取内容

输出字段：
- **skeleton**: 完整的投标文件组成结构，**原封不动复制原文**
  * 嵌套结构：`[{"title": "...", "items": [...]}, ...]`
  * 扁平结构：`["1.投标书", "2.开标一览表", ...]`
- **format_templates**: 格式模板的完整内容（如4.2所述）
- **source**: 来源说明
  * "detailed_list" - 找到详细的完整列表（投标文件组成/响应文件组成章节）
  * "compliance_table" - 从符合性审查表的实质性响应内容列提取多级目录结构
  * "requirements_structure" - 从采购需求/比选需求章节的多级目录结构提取
  * "outline_plus_inference" - 找到大纲，通过推断补充详细内容
  * "inferred" - 完全从评分因素推断

## 5. 采购需求详细内容（用于框架填充）

**提取目标**：
- **扫描全文**，从所有章节中提取对投标人有明确要求的内容，不限于"采购需求"章节
- 典型来源章节包括但不限于："采购需求"、"项目概况"、"项目需求"、"服务需求"、"技术需求"、"投标人须知"中的实质性要求等
- 按主题分类整理
- **保留原文**：尽可能保留招标文件中的原始文字
- **保留层级结构**
- **不遗漏**：任何对投标人提出的明确要求（如通识培训要求、专业模块要求、沉浸式研修要求、资源提供要求、后勤要求、跟踪指导要求等）都必须提取

**提取格式**：
```json
"requirements": {
  "topic_key": {
    "title": "主题标题（如'培训目标'）",
    "content": "招标文件中的原始文字内容",
    "sub_items": [
      {"title": "子项标题", "content": "子项内容"}
    ]
  }
}
```

# 注意事项

1. 如果文档中没有多包，packages数组留空[]
2. 如果文档包含多个包且所有包共用同一套评分标准（评分因素、分值、评分细则完全相同），scoring_factors只输出一份，并在packages同级添加 `"scoring_shared": true`；如果各包评分标准不同，则为每个包分别输出scoring_factors，并标注 `"scoring_shared": false`
3. 评分因素必须完整提取，不要遗漏
3. **skeleton必须原封不动复制原文结构**：
   - 不要重新分组（不要把扁平列表变成嵌套结构）
   - 不要重新编号（不要把"1.投标书"改成"（一）投标书"）
   - 不要遗漏任何项目
   - 如果原文是扁平列表就输出扁平列表
   - 如果原文是嵌套结构就输出嵌套结构
4. **format_templates**：模板文字必须完整提取（全文，不是摘要）
5. **requirements**：保留原文，保留层级结构
6. 所有分值必须是数字类型
7. 只输出JSON，不要有其他解释文字
8. **JSON转义**：所有字段值中如包含双引号，必须使用中文引号（\u201c\u201d）替代，避免破坏JSON格式。例如：填写"项目名称" → 填写\u201c项目名称\u201d
