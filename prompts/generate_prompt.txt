# 生成任务

请根据分析结果生成标书框架，遵循以下规则：

## 1. 骨架构建：格式模板章节优先

**核心原则**：招标文件通常有两处定义响应文件结构：
- **skeleton（9.2投标文件组成）**：概述性列表，告诉你响应文件有哪些部分
- **format_templates（第七章/投标文件格式）**：完整的格式定义，包含每一项的正确标题、编号、模板内容、表格格式、注意事项

**当format_templates存在时，以format_templates为准构建框架**：

### 1.1 章节标题以格式模板为准

skeleton可能写：
```
（1）资格及资信证明部分  ①响应函  ②供应商的资格及资信证明文件
```

但格式模板章节实际是：
```
封面格式(资格及资信证明部分)    ← 章节名
索引：一、响应函  二、供应商的资格及资信证明文件    ← 索引编号
一、响应函（正文）                                  ← 实际标题
二、供应商的资格及资信证明文件（正文）
```

框架应使用：
- 章节标题 → "资格及资信证明部分"（从封面/索引提取，不带skeleton的（1）编号）
- 子项标题 → "一、响应函"、"二、供应商的资格及资信证明文件"（格式模板的实际编号，不用skeleton的①②）

### 1.2 格式模板的每一项都必须出现

格式模板章节中的所有内容层级必须完整反映到框架中，不能只看skeleton的概括。

例如skeleton写"②供应商的资格及资信证明文件"，但格式模板中实际展开为：
```
二、供应商的资格及资信证明文件
  二-1 单位授权书
  二-2 证明材料
    二-2-1 资格承诺函
    二-2-2 资格证明材料
      - 营业执照等证明文件
      - 财务状况报告
      - 依法缴纳税收证明材料
      - 依法缴纳社会保障资金证明材料
      - 具备履行合同所必需设备和专业技术能力的声明函
      - 参加采购活动前三年内没有重大违法记录书面声明
    二-3 信用记录查询提示
    二-4 其他资格证明文件（若有）
```

**这些全部都必须出现在框架中**，不能概括为一句话。

### 1.3 扁平skeleton（无格式模板章节时）

如果没有format_templates，则直接使用skeleton作为骨架：
- 每项 → level 1 节点，保留原文编号和标题
- 不要自行分组

### 1.4 无明确skeleton时的通用格式兜底

**当分析结果中skeleton为空或source为"inferred"时**，说明招标文件中未找到明确的响应文件组成结构。此时按以下优先级构建框架：

**优先级1："符合性审查"/"符合性检查/采购需求"/"比选需求"/"服务需求"等章节中的多级目录**
- 如果分析阶段从"符合性审查"/"符合性检查/采购需求"/"比选需求"/"服务需求"等章节提取到了多级目录结构（source为"compliance_table"或"requirements_structure"）, 与评分因素进行*综合对比*后，以该结构为基础形成骨架
- 保留原文的多级编号和标题（如"1、基础管理 → 1.1角色层级"）
- 在此骨架基础上，补充评分因素对应的证明材料和技术方案节点

**优先级2：通用格式框架兜底**
- 如果连多级目录结构也没有（source为"inferred"），则使用以下通用响应文件框架作为骨架：
```
一、资格性文件
二、商务部分
  （一）响应供应商公司概况
  （二）类似项目业绩介绍
  （三）商务条款响应表
三、技术部分/服务部分
  （一）服务条款响应表
  （二）培训服务方案
    1.培训对象  2.需求分析  3.培训目标  4.培训主题
    5.培训时间  6.培训整体设计  7.培训课程表  8.培训方式
    9.考核评价  10.培训师资  11.实践基地  12.培训后勤保障
    13.研修平台保障  14.训后跟踪指导
  （三）培训管理方案
    1.组织机构与项目管理团队  2.培训进度流程管理
    3.培训质量保障措施  4.培训管理规章制度
  （四）安全应急预案
    1.安全管理措施  2.应急响应与处置方案
    3.网络研修平台危机应对  4.网络研修各种预警处理机制
四、价格部分
五、其他资料
```
- 通用框架中的子项可根据实际评分因素和采购需求进行裁剪：删除不相关的项，保留匹配的项
- 评分因素中有但通用框架中没有的项，追加到对应分类下

### 1.5 禁止事项

- **不得**自创标题或编号（标题和编号只能来自招标文件原文或通用格式框架）
- **不得**遗漏格式模板中的任何项目
- **不得**将多个子项概括为一句话

## 2. 章节封面和索引

**检查format_templates中是否有封面格式**：
- 如果有"封面格式(资格及资信证明部分)"等 → has_chapter_covers=true
- 为每个对应的一级章节生成cover_page和index_page
- 索引内容从格式模板的索引页提取，包含※注意事项
- 如果没有封面格式 → has_chapter_covers=false

## 3. 评分因素展开（核心）

**关键规则**：在format_templates中标记为is_free_form=true的节点下展开评分因素。

**操作步骤**：
1. 在format_templates中找到is_free_form=true的项目（通常是"供应商提交的其他资料"、"技术文件"、"项目实施方案"等）
2. 在框架中找到对应节点
3. 在该节点下，将评分因素作为子节点展开

**展开位置判断**：
- 评分因素应展开在哪个is_free_form=true的节点下，取决于该节点原文的描述
- 例如：如果某节点原文写"提供符合采购需求及评标方法和标准规定的相关证明文件"，则评分因素应在此节点下展开
- 例如：如果某节点原文写"技术和服务要求响应表"或"项目实施方案"，则技术方案类评分因素应在此节点��展开
- 不要自行假设展开位置，必须根据format_templates中各is_free_form节点的原文描述来判断

**展开规则**：
- 价格类因素（category为"price"）：不展开
- 技术类因素（category为"technical"）：展开为详细子项
- 商务类因素（category为"commercial"）：
  - 方案类（如"售后服务方案"）：在技术方案节点下展开
  - 证明类（如"业绩"、"专家队伍"）：在对应证明材料节点下展开

**评分因素子项展开**：
- 如果scoring_factors中某个因素的description或sub_items中包含明确的分项（如"包括但不限于：①参训人员；②培训方式；③理论依据；④学习工具"），必须将这些分项展开为下级子节点
- **复制而非剪切**：父节点的content必须保留完整的评分标准原文，子项展开是在父节点下额外添加子节点，不得因展开子项而删减父节点content中的任何文字
- 每个子节点的title为分项名称，content为该分项对应的评分要求原文
- 如果分项没有独立的评分要求描述，content填入该分项在整体评分标准中的对应部分

**采购需求内容归入框架**：
- analysis中requirements提取的全文明确要求，必须归入框架中最匹配的评分因素节点下作为content的补充
- **匹配优先级**：优先归入评分因素节点，只有确实无法匹配任何评分因素时才作为独立子节点
- **匹配判断标准**：需求内容涉及的主题词与评分因素的评审范围是否重叠
  - 涉及培训目标、课程设计、通识培训、专业模块、学习资源（文本图书、数字化课程）等 → 归入"培训目标和内容"
  - 涉及培训方式、沉浸式研修、线上线下、跟岗学习等 → 归入"培训方式"
  - 涉及调研分析、数据能力、参训人员分析、学习工具等 → 归入"重难点分析"及其对应子项
  - 涉及交通、食宿、餐费、保险等 → 归入"后勤保障"及其对应子项
  - 涉及训后跟踪、回访、学员发展报告等 → 归入"过程管理方案"的训后跟踪子项
  - 涉及自评报告、成果转化、经验提炼等 → 归入"培训质量管理方案"
- 如果某项要求确实无法匹配到任何评分因素（如纯资质要求、投标规则等），作为独立子节点放在is_free_form节点下

**编号格式**（展开部分使用标准层级编号）：
- level 2下：子节点编号用1. 2. 3.
- level 3下：（1）（2）（3）
- level 4下：① ② ③

## 4. 内容填充

**content字段填充规则**：

**核心原则**：模板文字就是正文，直接写入content，不加任何包装标记（不加"【模板原文：】"、"【要求：】"等标记）。

**段落对齐标记**：content中的每一行，如果在招标文件原文中有特殊对齐方式，必须在行首加标记：
- `[CENTER]文字` → 居中对齐（用于：函件标题如"投标函"、"承诺函"、"授权委托书"，"致：XX单位"等抬头行）
- `[RIGHT]文字` → 右对齐（用于：签章区域如"供应商：（全称并加盖单位公章）"、"日期：年 月 日"、"法定代表人：（签字或盖章）"等落款行）
- 无标记 → 默认左对齐（正文段落）
- 示例：
  ```
  [CENTER]投标函\n\n致：XX市公共资源交易中心\n\n根据贵方项目（编号：XXX）的招标文件，我方...（正文若干段）...\n\n[RIGHT]供应商：（全称并加盖单位公章）\n[RIGHT]日期：  年  月  日
  ```

**优先级1：模板文字（完整复制）**
- 在format_templates中找到与当前节点对应的模板
- 如果有full_text，**完整复制为content正文**
- **包含所有※注意事项**：模板下方的"※注意："内容必须一并复制到content末尾

**优先级2：表格结构（用标记包裹）**
- content中的表格部分必须用 `[TABLE_START]` 和 `[TABLE_END]` 标记包裹，使程序能识别并生成Word真实表格
- 标记内第一行为列头，后续行为数据行（如有预填数据），列之间用 ` | ` 分隔
- 表格前后的文字（表头信息、填写说明、※注意事项、签章信息）保持为普通文本，**不要**包含在标记内
- 示例：
  ```
  项目编号：\n货币及单位：人民币元\n\n[TABLE_START]\n采购包 | 投标报价 | 投标保证金 | 备注\n[TABLE_END]\n\n※注意：\n1、本表应按照下列规定填写：...\n\n供应商：（全称并加盖单位公章）\n日期：年月日
  ```
- 有预填数据行的表格示例：
  ```
  [TABLE_START]\n序号 | 商务条款 | 招标文件要求 | 投标人承诺 | 偏离说明\n1 | 付款方式 | 合同签订后...支付 | | \n2 | 服务地点 | 按采购人指定地点 | | \n[TABLE_END]
  ```

**优先级3：评分标准（完整复制）**
- 在评分因素展开的节点中，将scoring_factors的description**完整复制**到content
- **必须完整贴入评分标准原文**，不得缩写或概括，防止漏项

**优先级4：采购需求内容**
- 从requirements中匹配相关内容，直接写入content

**sub_items展开**：
- 如果requirements中有sub_items，展开为下级节点
- 每个sub_item也填充对应content

## 5. 格式模板子项的标题处理

- 格式模板编号（如"格式 7-1"、"格式 8-1"）在标题中**只保留序号**（如"7-1"、"8-1"），去掉"格式"二字
- 示例：`格式 7-1 供应商资格信用承诺函` → `7-1 供应商资格信用承诺函`

## 6. 特殊处理

- 标注"不适用"的项保留节点但content标注"【不适用】"
- 如果有notes，附加到content末尾
- 商务条件响应表、技术响应表等作为独立节点，不在其下展开评分因素

```json
      {"level": 2, "title": "1. 培训课程设计方案（技术评分20分）", "content": "（评分标准完整原文）", "children": []}
    ]}
  ]
}
```

**字段说明**：
- has_chapter_covers: 布尔值，是否有章节封面
- level: 层级（1-6）
- title: 标题（来自格式模板的原文标题和编号）。格式模板中的标题通常是居中的，必须加 `[CENTER]` 前缀，如 `"[CENTER]一、响应函"`、`"[CENTER]二、投标函"`。只有明确左对齐的标题才不加前缀
- content: 正文内容。格式标记：表格用 `[TABLE_START]...[TABLE_END]` 包裹；居中行用 `[CENTER]` 行首标记；右对齐行用 `[RIGHT]` 行首标记；缩进用空格（一级2空格，二级4空格）；无标记为左对齐。无内容则为空""
- cover_page: 封面页信息（仅一级节点有，has_chapter_covers为true时）
- index_page: 索引页信息（仅一级节点有，has_chapter_covers为true时）
- children: 子节点数组（即使为空也用[]）

**⚠️ title与content严禁重复（关键规则）**：

content是title之后的正文，绝对不能包含title中已有的文字。format_templates的full_text第一行通常就是标题（如"[CENTER]二、投标函"），这一行已经放入title字段，content必须从第二行开始。

```
✗ 错误示例（标题重复）：
  title: "二、投标函"
  content: "[CENTER]投标函\n\n致：采购人\n..."
  → Word渲染结果：先出现"二、投标函"（加粗），再出现"投标函"（居中）→ 重复！

✓ 正确示例（标题不重复）：
  title: "[CENTER]二、投标函"
  content: "致：采购人\n..."
  → Word渲染结果：只出现一次"二、投标函"（加粗居中），然后是正文
```

# 注意事项

1. **格式模板优先**：有format_templates时，框架的标题、编号、结构以format_templates为准，不以skeleton为准
2. **完整性**：格式模板中的每一项都必须在框架中出现，不能遗漏、不能概括
3. **内容完整复制**：模板正文、表格结构、※注意事项、编制说明，全部完整复制到content
4. **评分因素展开**：只在is_free_form=true的节点下展开
5. **content不为空**：尽可能填充模板文字或要求原文
6. **节点结构**：每个节点必须有level、title、content、children四个字段
7. **只输出JSON**，不要有其他解释文字。后续由程序读取JSON生成Word文档（宋体四号，标题加粗，正文不加粗）
8. **JSON转义**：content中如包含双引号，必须使用中文引号（\u201c\u201d）替代，避免破坏JSON格式
9. **Word排版意识**：title会渲染为加粗标题行，content会渲染为不加粗正文段落。确保title简洁（仅标题），详细内容放在content中
11. **严禁标题重复**：format_templates的full_text第一行通常是标题（如"[CENTER]二、投标函"），这一行放入title字段后，content必须从下一行开始，绝不能再包含标题文字。程序会先渲染title（加粗），再渲染content（不加粗），如果content重复了标题，Word中就会出现两遍
12. **格式模板标题默认居中**：格式模板章节中的各项标题（如"一、开标一览表"、"二、投标函"、"三．投标人资格声明书"等）在招标文件中几乎都是居中排版的，title字段必须加`[CENTER]`前缀。只有你确认截图中该标题是左对齐的，才不加前缀
10. **多包共用评分标准**：当analysis中 `scoring_shared` 为 true 时（即多个包共用同一套评分标准），只需生成一份框架，并在JSON顶层添加 `"applicable_packages": "第X-X包（评分标准相同）"` 标注适用范围；当 `scoring_shared` 为 false 时，需为每个包分别生成独立的框架
